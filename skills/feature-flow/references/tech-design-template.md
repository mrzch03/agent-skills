# 技术方案设计模版

> **用法**:按本结构**新建一篇方案文档**(工具无关:飞书/Confluence/Notion 均可),填好后链接挂回 issue。本文件是"必填清单"的真相源,实例写在文档里。
> **本模版与 `design-review-rubric.md` 一一对应**——照它写,就能过评审。**带 \* = 必填**;标 [serve]/[前端] 的按方向填。
> **评审人**:项目开发 + code review 同学 + 测试同学。

## 一、背景与目标
- **项目背景 \***:当前背景、需求 wiki 链接。
- 名词解释:新概念 / 名词。
- **解决目标 \***:要达成什么(是目标不是方案);效果目标 / 量化指标。
- **业务功能 \***:要实现的功能,list 罗列。

## 二、方案设计
- **方案选型(可选)**:有备选时给 ≥2 候选 + 差异性对比 + 设计折中——为什么选它、舍了什么;无显著备选可不写。
- **业界方案对比(可选)**:相关业界/legacy 技术方案怎么做的(符合业内标准)。
- **重难点分析 \***:把最难/最关键的拎出来 + 解决方案(别平铺)。
- **整体/系统设计 \***:**架构设计图 / 功能流程图 / 时序图**(清晰明了)。
- **交互流程**:业务接入交互流程 / 用户操作访问交互流程(图清晰明了)。
- 功能 / 模块说明。
- **数据库设计 \*** [serve]:新增/修改表、需优化的 SQL。⚠️ **变更兼容 2 个版本**;大表每条 SQL 过 EXPLAIN(`type≠ALL`、选择性索引)。
- **接口设计 \*** [serve]:接口列表 / 地址;接入外部附链接。
- **组件抽象与划分 \*** [前端]。
- 外部依赖:依赖哪些团队 / 资源。

## 三、可行性(不假设,带证据 —— dev-standards 判据 8)
- **可行性论证 \***:资源 / 成本 / **可扩展性 / 鲁棒性**;关键"行得通"的断言附 **POC / 实测 / 事实 / 先例**,够不到的环境标"未证"。
- **性能影响、吞吐量估计** [serve]。
- **验证方式 \***:怎么验证问题被解决,**并说明是否影响其它模块**。

## 四、交付与风险
- **上线方案 \***:准备 checklist + 发布方案(依赖哪些服务、先后顺序)。
- **回滚方案 \***:回滚具体步骤。⚠️ **兼容性 \***:DB 变更兼容 2 个版本。
- 灰度方案 / 监控方案。
- **阶段性成果**:里程碑拆分。
- **验收标准 \***:怎么算做完。
- **风险 \***:已知风险 + 应对。
