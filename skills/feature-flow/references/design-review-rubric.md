# 技术方案评审规范(第3步「评审方案」的依据)

评审人:项目开发 + code review 同学 + 测试同学。对照下面**原则 + 评定项**逐条过,给出 **A/B/C/D** 评分与结论。

## 评定原则(方案整体看这 7 条)
1. **目标、背景、方案明确**。
2. **可行性**(可扩展性 + 鲁棒性)。
3. **突出重、难点**(别平铺,把最难/最关键的拎出来)。
4. **方案多选 + 差异性对比 + 设计折中(可选)**:有备选时给对比、讲清为什么选它/舍了什么;无显著备选可不写。
5. **全局思维 / 架构全局认知**(放进系统整体看,不是孤立模块)。
6. **友好性**(实现成本 / 接入成本)。
7. **符合业内标准**(有惯例用惯例,不自创)。

## 评定项(逐条核对;标 [serve]/[前端] 的按方向适用)
- [ ] 方案背景、解决目标、名词解释、技术方案阐述**清晰**。
- [ ] **架构设计图 / 功能流程图 / 时序图** 清晰明了。
- [ ] **可行性**(资源 / 成本 / 扩展性)——附**证据**,不假设(呼应 dev-standards 判据 8)。
- [ ] 业务接入交互流程 / 用户操作访问交互流程,图清晰明了。
- [ ] **关键问题 / 难点分析 + 解决方案**。
- [ ] 功能 / 模块说明。
- [ ] **对解决问题提供验证方式**,并说明**是否影响其它模块**。
- [ ] **可行性分析**;业界相关技术方案对比(可选)。
- [ ] [serve] 性能影响、吞吐量估计。
- [ ] [serve] 接口设计、数据库设计。
- [ ] [前端] 组件抽象与划分。
- [ ] **上线方案、回滚方案**。
- [ ] 阶段性成果。
- [ ] **验收标准**。
- [ ] **风险**。

## 评分(评审产出:打分 + 结论 + 待办)
- **A —— 优秀**:可**立即分配资源执行**。
- **B —— 良好**:需继续协商资源或**完善技术细节再执行**。
- **C —— 合格**:能满足目标 / 系统要求,但**存在风险或未考虑后续扩展性**,细节有待完善。
- **D —— 不合格**:整体阐述不够明确 / 解决方式不合理 / 部分设计有功能或逻辑漏洞,**需再完善后再次评审**。

> **建议进开发闸门(可调)**:A 直接进;B 补齐细节/资源后进;**C、D 回炉**(补风险与扩展性 / 重新设计)再评审。评分与待办回贴到 issue。
