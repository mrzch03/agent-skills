# scenario-map — 能力总入口

拿到一个任务先做两件事:**① 定环境(prod / test)② 按场景路由到对应 skill**。各 skill 高内聚、用完即丢;知识一份一个家,别跨 skill 重抄。

## 核心原则(动手前过一遍,所有场景通用)
- **第一性原理审题,优雅方案直击本质**:很多"问题"是上游设计缺陷的症状——修设计让问题**蒸发** > 技术上"解决"。新问题按序自问:①是不是上个补丁的债 ②需求本身合理吗(需求是输入不是圣旨,带来不成比例复杂度的要挑战)③是不是设计缺陷的症状 ④能不能删掉一个东西让问题不存在 ⑤确需新增,用最简模型卡它。
- **删机制优于加机制**;**简洁链路 / 行业标准 / 不过度封装(抽象等第二个使用者)/ 以少为多**。
- 排障铁律:**先定环境**(用约定的测试种子号区分 test,真实客户号=生产)。**容量报错先问"这字段本该存吗"**(存错 > 存不下)。
- 详细规范(流程/提交/DB·Redis/协作)→ **dev-standards** skill。

## 路由表(场景 → 用哪个 skill)— 模板

> 这是**模板**:右列按你的项目填成实际 skill 名。下面给的是常见场景的占位示例。

| 你要干什么 | 去哪个 skill |
|---|---|
| **做需求 / 开发新功能**(有 DB/接口依赖,完整流程) | `feature-flow`(挑战→issue→技术方案→评审→开发→codex→测试) |
| **排查线上问题**(查库/日志/延迟/历史 case) | `<你的排查/知识库 skill>` |
| **某业务领域的活**(该领域的全链路 playbook) | `<你的项目认知 skill,如 clone-anchor-unify 那种>` |
| **开发规范**(设计原则/流程/提交/DB·Redis/协作) | `dev-standards`(被所有场景 skill 引用) |
| 开 issue / 建带编号分支(issue 优先) | `<你的 issue 流程 skill>` |
| 提交 / 建 PR·MR | `<你的提交流程 skill>` |
| codex 独立评审本地分支("走流程") | `<你的 review skill>` |
| 跑应用验证 / 自测出证据 | `verify` · `run` |
| 硬 bug / 性能回归 的纪律化诊断 | `diagnose` |

> 通用工具 skill(handoff / write-a-skill 等)按 description 自动触发,不在此一一列。

## 维护
- **新增业务场景 skill 时,在本路由表加一行**(场景 → skill + 一句话)。
- 场景 skill 只放本场景 playbook + 引用 dev-standards;通用规范别下沉到场景 skill。
- 交接:每个场景 skill 带薄 `progress.md`(git 表达不了的待验证/弯路)+ 会话末 `/handoff` 写续作文档。
