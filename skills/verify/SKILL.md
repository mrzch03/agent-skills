# verify — 带证据的验证

"编译过 / 单测绿"≠"做完了"。验证 = **真环境跑 + 观察预期行为 + 留可复核的证据**。

## 怎么做
1. **跑**:把改动真跑起来(应用/服务/脚本),走到要验证的那条链路。
2. **观察预期跳变**:先说清"应该看到什么"(状态 A→B、某日志出现、某行写入、某指标变化),再逐条核对。
3. **留证据**:日志片段 / 截图 / DB 行 / trace id / 命令输出——挂回 issue,让没看过过程的人也能复核。
4. **证伪**:回归测试要"被违反就 FAIL"——把 bug 加回去验证用例真的红,证明它测的是这件事,不是同义反复。
5. **诚实标边界**:够不到的环境(如生产)别用测试结果冒充,标"**测试已证 / 生产未证 + 怎么证**"。

## 和 dev-standards 判据 8 一脉
"方案必带可行性证明,不假设"——**验证就是给出那份证明**。没证据的"应该可以"不算数。
