# 设计原则(方案评审先过这关,再谈实现)

## 总纲:三层审题,从最上层挑战起

拿到一个"问题",第一性原理逐层问——**问题在哪层出生,就在哪层消灭**:

1. **需求层:这个需求本身合理吗?** 需求也是设计出来的,也会有缺陷,它是输入不是圣旨。信号:需求带来不成比例的复杂度;需求内部自相矛盾;需求描述的是"方案"而不是"目标"(X-Y 问题)。挑战方式:回到"用户到底要什么",提替代需求——**挑战归挑战,改不改由用户拍板**(协作分层)。
   案例:一条"某标识不得重复"的需求催生了两轮方案返工,追到底发现该标识本来就允许重复——需求本身是误传;一条"两种状态永不混用"的约束,一句话就简化掉了一整类复杂度。
2. **设计层:是不是设计缺陷漏下来的症状?** 是 → 修设计,问题蒸发(连同兄弟问题一起)。
   案例:一类反复出现的播放故障。游标、整窗重推、"多取 N 条"全是对症状的技术解决;直到把"队列即时间线"这个正确模型立住,补丁全删,问题不再存在。
3. **技术层:确认是真技术问题,才轮到技术方案。**

判错层的代价:在下层"解决"上层的缺陷,补丁把缺陷固化成债,之后每个新症状都要再打补丁——机制堆叠就是这么长出来的。

## 八条判据

1. **直击本质**:方案的衡量标准不是"能不能跑",是"它对付的是不是问题本身"。写之前能一句话说出"根因是 X,所以做 Y";说不出就还没想清。
2. **简洁链路**:数据流一条线讲得完;链路每多一个环节,排障成本翻倍。案例:录制收尾三轮演进(段计数对账 → 队列空判定 → 末句标记),越改越短,最后的最对。
3. **行业标准实现**:有惯例用惯例——标准库、既有契约、legacy 行为都是惯例。自创格式/协议/算法 = 未来的对齐成本。案例:给某 ID 自创前缀格式,违反既有契约约定的格式,卡死下游消费方。
4. **不过度封装**:不为单一实现配 interface;不加 DTO 映射层;抽象等第二个使用者出现再做。
5. **以少为多**:遇到边界情况先问"能不能删掉一个东西让这个问题根本不存在";保护层是最后手段,每个机制都是永久维护负担。案例:一套收尾判定机制(总数统计+分段计数+显式停止)加了又整体删除,换上一个"末项标记"后全是冗余。
6. **新概念过不变量**:每个新增概念必须被核心不变量(单一真相源 / 会话自包含 / 单向数据流)接纳;接纳不了,先怀疑概念,不是改不变量。
7. **单文件 ≤ 500 行(新代码硬约束)**:新增/重写的源文件超 500 行就**按职责拆**(不是机械截断)。信号:一个文件管了多件事、读一遍要翻半天、改一处怕碰另一处。与"不过度封装"配套——拆是为可读/可测,不是为拆而强抽 interface。已超标的存量文件不强制重构,但**新代码不准再叠上去**。
8. **方案必带可行性证明,不假设**:提方案就同时给"它行得通"的证据——最小 POC / 真环境实测 / 查到的事实 / 已存在的先例。凡"应该可以 / 估计 / 默认如此 / 按经验"= 未验证假设,**要么去验证,要么显式标"未证"** 让评审知道风险在哪。够不到的环境(如生产)别用测试结果冒充。判错的设计 + 没验证 = 加倍返工。

## 使用方式

方案讨论时把这八条当 checklist 过一遍;被否的方案记 progress.md 弯路,可泛化的进 lessons/heuristics.md。
